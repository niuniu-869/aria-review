# 开源体验修复开发计划（2026-07-02）

## 背景与方法

开源为 aria-review 后反馈：整体用户体验不佳、功能耦合差、文献搜索/导入存在 bug。本计划基于 4 条并行 codex 深度审计（搜索导入链路 / 前端逐页 UX / 跨服务耦合契约 / 新用户首跑体验），共产出 ~37 条发现；全部 P1 与高级别发现已由 Claude 逐条对照代码核实，剔除 2 条误报（导入弹窗残留旧结果——onClose 已有 `importMut.reset()`；根目录 png 拖累公开仓——已被 `/*.png` gitignore，公开仓无此问题）。

**执行协议（沿用前端统一升级里程碑协议）**：每个任务卡交 codex 开发 → Claude 跑单测 + E2E/Playwright 真实链路验收 → `codex review` 无 P1 → commit（按既往里程碑协议每卡验收后提交；正式开跑前向用户确认一次提交授权）。每阶段结束整体回归一次。**凡改动 API 响应的任务卡（P1-2/P1-3/P2-6 等），必须在同卡内同步 `packages/contracts/openapi.yaml` 并重新生成 `schema.d.ts`（`pnpm gen:api`），不等 P3-1。**

规模标记：S ≤ 半天，M ≈ 1 天，L ≈ 2-3 天（codex 执行口径）。

## 里程碑顺序与依赖

```
P0（并行，~2天）
   └→ P1-1/P1-2 先行（数据正确性 + 失败可见后端半）
        └→ P2-1/P2-2/P2-3（依赖 P1-2 的 latestCorpus 契约）
             └→ P2 其余 → P3-1→P3-2→…→P3-6（顺序）→ P4 穿插
```

- **P1-2 ↔ P2-1/P2-2 是同一个用户故事**（「构建失败后用户能看到原因并重试」），必须在同一里程碑内联调收口，单独上任何一半都不产生用户价值。
- 契约变更随卡走（openapi.yaml + `pnpm gen:api`，见执行协议），P3-1 只负责补齐存量缺口与 CI 防漂移。
- 每阶段结束跑全量：`pytest`（注意 allow_real_llm_router 标记会真跑 LLM）+ 前端 `pnpm test` + Playwright E2E + 20033 部署冒烟（维护者环境步骤：按既有部署方式重启三服务后，curl 首页 200 + 走一次「检索→导入→构建→概览」happy path）。
