# 0.6.2 · 体验止血与漏斗疏通方案

日期：2026-07-10 ｜ 依据：生产库 bibliocn 全量行为数据 + nginx/journald 日志 + 代码核查 ｜ codex 二审：session 019f4bdd-257e（已采纳裁决）

## 0. 一句话诊断

技术全部跑通（18/18 agent run done、综述 46 秒出稿），但真实用户在三处硬墙上流失：**计量分析 500 崩溃、空库空跑 + Agent 越权承诺、全文墙无路标**；跨天回访 1/12，产品还没让第 13 个用户"首次使用就走到一个可信的成功结果"。

### 1.2 三个流失现场（逐条有日志实证）

**用户 16（xiong_zhaoyang@163.com）——被 500 杀死的次日旗舰候选**
07-06 19:43 连跑 2 次检索入库 70 篇 → 19:46 打开计量分析 sources 页 → 连续 4 次 HTTP 500 → 流失，再未回访。
根因：`GET /projects/{id}/corpus/{cid}/sources` 的响应模型 `HSourceItem.source: str` 拒绝 R 服务返回的 `null`，FastAPI `ResponseValidationError` 直接 500（journald 07-06 19:46:31 有完整栈）。同项目 04 日还有 28 次 overview 502（r 容器部署窗口，QA 流量）。

**用户 17（2353182@tongji.edu.cn）——空库裸奔 + 越权承诺**
07-08 建项目后 0 文献，直接从 GAP 入口要综述 → agent 真跑了一轮才回复"0 篇无法生成"（浪费一轮 LLM）；`final_output` 开头泄漏内心独白（"我需要先告知用户现状…不过，用户要求直接生成综述。我先确认一下…"）；回复承诺"检索后您可以纳入语料库"——但 `entries.py` 中 `ENTRY_GAP` 工具白名单只有 `read_paper/scratchpad/search`，**无 project/library/review 工具，承诺无法兑现**。用户流失。

**用户 18（2580237058@qq.com）——全文墙上的 8 次撞击（唯一被验证的"卡点→转化"）**
07-09 检索入库 78 篇 → 18:44–18:52 被综述 precheck `no_fulltext` 挡 **8 次 / 8 分钟**（埋点实录）→ 自行摸到文献库全文回填 → 18:53 起 46 秒出综述，当天连生成 master/phd 两份。结论：**旗舰体验本身能打动人，卡点在"从被挡到补全文"之间没有路，全靠用户自己摸**。

### 1.3 其余硬伤

- **feasibility-scout（0.6 P2 旗舰）前端零接线**：后端 `POST /gaps/{id}:feasibility` 存在，`grep feasibility apps/web/src` 除 Welcome 营销文案外零命中；生产 `feasibility_verdict` 0/46。**Welcome 页在宣传"新颖度 × 可行性双重裁决"，产品里点不到**——承诺失真。
- **时间戳时区分裂**：`ai_job.created_at` 用 `func.now()`（DB 本地 +8），`repositories/ai_job.py:85` 写 `completed_at` 用 `utcnow()`，生产 6 条 review 时长全为 **-8 小时**，时长类指标全废。
- **埋点仅覆盖 ReviewPanel 4 事件**：注册/登录/建项目/检索/入库/GAP/回访全盲区，回访率只能从业务表反推。
- **credit_ledger 0 条消费**：计费从未扣费（仅 2 条赠送）。计量闭环未接。
- **gap 核验输出是死胡同**：5 条 verified 里 3 条 likely_filled（伪空白）、2 条 inconclusive，只给结论不给下一步。
- **ai_job 无重试**：1 条 gap_verify 因 Sciverse ConnectTimeout 整 job fail（当前连通性已恢复，属瞬时故障但无韧性）。

## 2. 方案（codex 二审后定稿）

统一组织原则（codex 指出的最大漏项）：**项目就绪度状态机**。前端已有全部原料（`includedCount`、`readableFulltextCount`、`fulltextEligibleCount`…），做一个共享 selector：

```
无文献 → 有题录 → 有可读全文 → 已纳入全文 → 可综述/可GAP
```

P0/P1 各项的"门禁"和"下一步推荐"全部从这一个 selector 读，不各写各的（DRY）；不建新框架、不加后端状态（YAGNI）。

### P0 · 止血（1–2 天，全部是"正在让真实用户失败或失信"的事）

| # | 动作 | 落点 | 验收 |
|---|---|---|---|
| P0-1 | R 代理响应模型**全面** null 容错（不止 `HSourceItem.source`——`SourceItem/BradfordItem/HAuthorItem` 等所有 R 数据喂养的必填 `str` 一律 `str \| None` + 前端渲染兜底） | `services/agent/app/schemas.py`；前端对应渲染组件 | 用含 null source 的 R 返回打 sources/authors/documents 全 200；回归测试固定 null 样本 |
| P0-2 | chat 入口空库门禁：GAP/综述 chat 入口在语料不满足时，输入框上方常显就绪度卡（复用 `GapReadinessCard` 视觉），**发送前拦截**，不再让 agent 空跑一轮 | `ChatWorkbench` / `ResearchView`，读共享 readiness selector | 空库项目在 gap 入口发消息 → 前端拦截并给最短下一步，`agent_run` 零新增 |
| P0-3 | Agent 承诺边界：三入口 persona 明确"只承诺本入口工具能做的事"，越权需求 → 指路对应入口（带链接文案）；同时 `final_output` 后处理剥离内心独白前缀 | `agent/prompts.py`（GAP_SYSTEM 等）；`run_controller`/engine 对 final_output 做独白剥离 | 空库 gap 入口重放用户 17 场景：回复无独白、无"帮你纳入/生成综述"类越权承诺、含指路链接 |
| P0-4 | precheck 卡内嵌**一键补全文**：`no_fulltext` 卡片上直接放"自动补全文"按钮（复用 `useBackfillFulltext` mutation，幂等、不改纳排状态），就地进度 + 完成后自动刷新 precheck | `ReviewPanel.tsx` + `useReviewJob.ts` | 重放用户 18 场景：从被挡到可生成 ≤ 30 秒、零页面跳转；埋点 `review_precheck_blocked → backfill_click → review_generate_click` 链路可查 |
| P0-5 | Welcome 承诺对齐：撤下/改写"可行性双重裁决"文案（一行改动），待 P1-1 接线后再恢复 | `WelcomePage.tsx` | 线上 Welcome 不再宣传产品内不可达的能力 |

### 不做（本轮明确排除）

- **邮件召回系统**：12 人样本不值得建设，先人工触达 5 个沉睡注册用户。
- **大改 onboarding**：5 个未建项目用户中 4 个是 07-04 老版本注册（当时无 hero/tour），样本污染——暂缓而非证明没问题，P1-2 埋点铺开后用新数据再判。
- **StageBar 加节点 / 新增顶层入口**：0.6.1 codex 已裁决过为 YAGNI，维持。

## 3. 12 用户样本下的结论边界（codex 裁决，防过度推断）

- 回访 1/12 只说明早期留存危险，**不能**推出"召回无效"或"核心价值不成立"。
- gap 3/5 伪空白**不能**证明 GAP 功能没价值——语料质量、检索式、领域都可能主导。
- credit 0 消费只证明计费未闭环，不代表付费意愿。
- 唯一可高置信行动的信号：三个流失现场（有逐条日志）+ 用户 18 的转化路径。

## 4. 发布后看什么数（0.6.2 验收指标）

1. `review_precheck_blocked` → `review_generate_click` 转化时间中位数（目标 < 1 分钟，基线 8 分钟）。
2. 空库 agent_run 数（目标 0，基线：用户 17 场景）。
3. corpus 分析端点 5xx 数（目标 0，基线 4 次/真实用户会话）。
4. `feasibility_verdict` 非空数（目标 > 0，基线 0/46）。
5. 埋点铺开后首个完整周的 D1 回访率（建立真基线，当前 8% 是脏口径）。
