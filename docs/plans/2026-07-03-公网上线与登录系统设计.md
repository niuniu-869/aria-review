## 0. 目标与约束

**目标**：让 `https://aria-review.com` 可对外访问，且未登录者无法接触任何数据或 AI 能力。

**硬约束**（决定这不是"加个登录框"那么简单）：

| 事实 | 依据 | 影响 |
|---|---|---|
| 后端零认证 | `app/main.py` 所有路由仅 `Depends(get_r_client)`，唯一边界是 `CORS_ORIGINS`（`config.py:34`） | 认证是必答题 |
| 多租户半成品 | `Paper.owner_id` / `Project.owner_id` 已预留（`models.py:64,167`，nullable+index），但**无 User 表、无任何 owner 过滤/写入代码** | 隔离是选择题 |
| `Project.name` 全局唯一 | `uq_project_name`（`models.py:172`） | **多用户下硬冲突**——必须改造 |
| 依赖 DeepSeek/MinerU 付费 API | `config.py:43-62` | 需决定成本承担 |
| BYOK 已就绪 | `llm.py:override_from_key/resolve_llm_params`，key 仅内存透传不落库 | 零成本上线的抓手 |
| 安全底子好 | `net_safety`（SSRF）、gitleaks、SECURITY.md、私有 base 拒绝 | 可复用 |
| 数据库口令硬编码 | compose `bibliocn:bibliocn`（docker-compose.yml:84） | 公网前必须改 |

## 1. 三个必答决策（决定架构量级）

### 决策 A — 系统定位
| 选项 | 数据模型改动 | 工作量 | 适用 |
|---|---|---|---|
| **私人门禁**（共享一个工作区） | 零 | ~1-2 天 | 先低调上线、给评委/合作者演示 |
| **邀请制多租户**（各自数据 + 邀请码） | User 表 + owner_id 全链路 + 改 name 唯一约束 | ~1 周 | 小范围真实用户 |
| **开放注册 SaaS** | 上者 + 邮箱验证/配额/计费/合规 | ~2-3 周 + 持续运维 | 公开产品 |

### 决策 B — AI 成本
| 选项 | 改造 | 代价 |
|---|---|---|
| **BYOK 用户自带 key** | 最小（复用 `override_from_key`），加"用户 key 加密存储" | 你零成本，用户门槛高 |
| **平台供给 + 配额** | 加用量计数/限流/账单 | 你承担成本，需盯账单 |
| **先只开离线/demo** | 零 | 最稳过渡，付费功能需 BYOK 解锁 |

### 决策 C — 登录方式（可叠加）
邮箱+密码 / 第三方 OAuth / 邀请码-预置账号 / 反代层统一认证（Authelia/Basic Auth）。

**推荐组合（分阶段落地）**：Phase A 反代层 Basic Auth 应急上线 → Phase B 应用层邮箱密码+邀请码（承载数据归属）→ Phase C 加 OAuth + 开放注册。

## 2. 目标架构

### 2.1 认证分层（两层各司其职）

```
Internet ──TLS──▶ Caddy (aria-review.com)
                    │  · Let's Encrypt 自动证书
                    │  · 安全响应头 (HSTS/CSP/X-Frame-Options)
                    │  · 全局限流 (登录爆破/滥用)
                    │  · [Phase A] Basic Auth 前置门禁
                    ├──▶ /api/*  → agent:8000 (FastAPI)
                    │              · [Phase B+] 应用层会话 = 真正的 user 身份
                    │              · 数据按 owner_id 隔离
                    └──▶ /*      → web (静态 SPA, nginx)
```

**为何认证主体放应用层而非纯反代层**：反代层 Basic Auth 无法把请求映射到 `user_id`，做不了数据隔离与 BYOK 的 per-user key。反代层只做"粗门禁 + TLS + 限流 + 头"，应用层做"谁是谁 + 数据归谁"。

### 2.2 会话机制选型

推荐 **server-side session 存 Postgres + httpOnly Cookie**（而非纯 JWT）：
- 可即时吊销（改密码/踢下线），小规模无横向扩展压力；
- httpOnly + `Secure` + `SameSite=Lax` 防 XSS 窃取、缓解 CSRF；
- 无需引入 JWT 密钥轮换复杂度。
- 若未来要多实例水平扩展再迁 JWT/Redis。

密码哈希：**argon2-cffi**（首选）或 bcrypt（passlib）。

### 2.3 数据模型改造（Phase B 核心）

新增表：
```python
class User(Base):
    __tablename__ = "user"
    id: Mapped[int] = _pk()
    email: Mapped[str] = mapped_column(String(255), unique=True, index=True)
    password_hash: Mapped[str | None] = mapped_column(String(255), nullable=True)  # OAuth-only 用户为 null
    display_name: Mapped[str | None] = mapped_column(String(120), nullable=True)
    role: Mapped[str] = mapped_column(String(16), default="user")   # user / admin
    status: Mapped[str] = mapped_column(String(16), default="active")  # active / disabled
    # BYOK: 用户 API key 用应用级 Fernet 密钥加密后存储，取用时解密透传
    encrypted_keys: Mapped[dict | None] = mapped_column(JSON, nullable=True)
    created_at: Mapped[dt.datetime] = mapped_column(server_default=func.now())

class Session(Base):
    __tablename__ = "session"
    id: Mapped[str] = mapped_column(String(64), primary_key=True)  # 随机 token 的 sha256
    user_id: Mapped[int] = mapped_column(ForeignKey("user.id", ondelete="CASCADE"), index=True)
    expires_at: Mapped[dt.datetime] = mapped_column(index=True)
    created_at: Mapped[dt.datetime] = mapped_column(server_default=func.now())

class InviteCode(Base):   # 邀请制阶段
    __tablename__ = "invite_code"
    code: Mapped[str] = mapped_column(String(40), primary_key=True)
    used_by: Mapped[int | None] = mapped_column(ForeignKey("user.id"), nullable=True)
    created_at: Mapped[dt.datetime] = mapped_column(server_default=func.now())
```

改动既有表（**breaking，需 alembic 迁移 + 数据回填**）：
- `Project.owner_id` → NOT NULL + FK(user.id)；**删除 `uq_project_name`，改 `UniqueConstraint("owner_id","name")`**（每用户内唯一）。
- `Paper.owner_id` → 用户私有库 owner_id NOT NULL；公共 demo 语料保留 owner_id IS NULL（现有部分唯一索引 `uq_paper_dedup_null_owner` 正好兼容）。
- 回填：上线前把存量 project/paper 归属到一个"迁移账户"，避免 NOT NULL 迁移失败。

### 2.4 越权防护（最大风险点）

**原则（SOLID-D + DRY）**：不在每个路由手写 owner 过滤（漏一处即数据泄露），而是：
1. 一个 FastAPI dependency `get_current_user` 从 cookie 解析 session → user；
2. repository 层函数**强制**收 `owner_id` 参数，查询恒带 `WHERE owner_id = :uid`；
3. 加一条测试：随机 user B 访问 user A 的 project_id 必得 404（不可 leak 存在性）。

```python
# app/auth.py（新增）
async def get_current_user(request: Request, s=Depends(get_session)) -> User:
    token = request.cookies.get("aria_session")
    user = await session_repo.resolve(s, token)   # 校验存在+未过期
    if not user or user.status != "active":
        raise ApiError(401, "unauthenticated")
    return user
```

### 2.5 BYOK（复用现有透传）

现状：`override_from_key(api_key=...)` 已支持请求内透传。改造：
- 用户在 Settings 填 key → 用 **Fernet(应用密钥)** 加密存 `User.encrypted_keys`；
- 请求时解密 → 传给 `override_from_key`，**永不回显明文、永不日志**（延续 `llm.py` 现有"永不记录 Authorization"约定）。

## 3. 分阶段路线图

### Phase A — 反代层门禁，今晚可上线（~0.5-1 天）
**目标**：`aria-review.com` 有 TLS，未认证者进不来。**零应用改动**。
- Caddy 配置域名 + 自动证书 + 反代到现有 8000/8080；
- Basic Auth（或 Authelia，若要登录页）保护全站；
- 改 Postgres 强口令、确认 5432 不对公网暴露（compose 已 `127.0.0.1` 绑定，保持）；
- BYOK 已可用（用户在前端设置填 key）。
- **验收**：`curl https://aria-review.com` 未带凭据 → 401；带凭据 → 200；`curl http://<ip>:8000` 从公网不可达。

```caddy
# /etc/caddy/Caddyfile
aria-review.com {
    encode gzip zstd
    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
        X-Frame-Options "SAMEORIGIN"
        X-Content-Type-Options "nosniff"
        Referrer-Policy "strict-origin-when-cross-origin"
    }
    @api path /api/*
    handle @api {
        reverse_proxy 127.0.0.1:8000
    }
    handle {
        reverse_proxy 127.0.0.1:8080   # SPA
    }
    # Phase A 应急门禁（Phase B 上线后移除，改由应用层认证）
    basic_auth {
        aria $2a$14$<bcrypt-hash>   # caddy hash-password 生成
    }
    # 登录接口放行更严限流交给应用层；此处全局兜底
}
```

### Phase B — 应用层认证 + 邀请制多租户（~1 周）
- 新增 `User/Session/InviteCode` 表 + alembic 迁移 + 存量回填脚本；
- `app/auth.py`：注册（校验邀请码）/登录/登出/`get_current_user`；argon2 哈希；
- 全部业务路由挂 `Depends(get_current_user)`；repository 强制 owner 过滤；
- 改 `Project` name 唯一约束为 per-owner；
- 前端：登录/注册页 + 401 拦截跳登录 + Settings 填 BYOK key；
- BYOK key 加密存储（Fernet）；
- **越权测试**：user B 读 user A 资源 → 404；覆盖 project/paper/corpus/agent_run 全部读写路径；
- 移除 Caddy Basic Auth（改由应用层接管）。
- **验收**：两个账户互不可见对方数据；happy path（注册→建项目→上传→综述）端到端真跑通。

### Phase C — 开放注册 SaaS（~2-3 周 + 运维）
- 邮箱验证（发信：Resend/SES）+ 找回密码；
- OAuth（Google/GitHub，`authlib`）；
- 限流/配额（`slowapi` 或 Caddy `rate_limit`）+ 用量计数（若平台供给 key）；
- 合规：隐私政策 / ToS / 数据删除；备份（pg_dump 定时）；监控（健康探针告警）；
- `pip-audit` 进 CI（已有 gitleaks + openapi-drift）。

## 6. 风险与开放问题
- **越权是头号风险**：owner 过滤漏一处即跨租户泄露 → 必须 repository 层统一 + 全路径测试。
- **`uq_project_name` 迁移**：存量数据回填 owner 前不能加 per-owner 约束；需先回填再换约束，顺序敏感。
- **BYOK 体验**：用户没 key 则 AI 功能不可用，需前端清晰引导 + 保留零 key demo 兜底。
- **成本失控**（若选平台供给）：无配额=账单风险，Phase C 前不建议开放平台 key。
- **合规**（公开注册）：用户上传 PDF + 账户数据涉隐私，需 ToS/隐私政策/删除权。

## 8. Codex 二审补充与修订（2026-07-03）

二审确认主方向成立，但指出我**低估了越权治理与滥用控制**。以下为对前文的实质修订（优先级高于"选 session 还是 JWT"这类选型）。

### 8.1 会话硬化（补 §2.2）
- Cookie 存随机 token，**DB 只存 `sha256(token)`**，不存明文。
- 登录后**轮换 session**；改密码/禁用用户时可批量吊销。
- `SameSite=Lax` 不是完整 CSRF 防护：**所有非 GET/HEAD 请求校验 `Origin`/`Referer` 属于 aria-review.com**，再叠加 CSRF token。
- session 表加 `last_seen_at` / `ip_hash`（排查异常）。

### 8.2 IDOR 工程化（重写 §2.4，头号修正）
风险不在 `project`/`paper` 两张主表，而在**没有 owner_id 的子表**：`draft`/`ai_job`/`agent_run`/`artifact`/`corpus`/`corpus_paper`/`attachment`/`paper_extraction`/`agent_event`/`tool_invocation`/`document_structure` 全靠外键**间接**归属，任一 join 漏判即穿透。**别只靠人肉在每个 query 加 `WHERE owner_id`。**三层强制：
1. **路由层**：统一 dependency `get_owned_project(project_id, current_user)` → 返回已验证属主的 Project 对象；子资源操作只接受该对象，禁止裸 `project_id`。
2. **repository 层**：按**作用域对象** `ProjectScope(user_id, project_id)` 组织，子表查询全部从 scope 派生（前置已验属主），而非散收 `owner_id`。
3. **测试**：不是"一条 404"，而是**权限矩阵**——覆盖 project/paper/project_paper/corpus/draft/ai_job/agent_run/artifact/attachment-PDF/report-download/**流式 job/异步 polling/导出** 每类入口，user B 访问 user A 一律 404（不泄露存在性）。
- **Postgres RLS**：最强安全网，但多数表无 owner_id、policy 要 join 推导，复杂易错 → **列为 Phase C 防线，不在 Phase B 首版全量上**。
- **SQLAlchemy 全局 query filter**：raw SQL/alias/关系加载易绕过 → **不作主防线**。
- 取舍结论：**应用层统一 scoping + 权限矩阵测试先落地；核心表后续再补 owner_id 或 RLS。**

### 8.3 迁移安全（补 §2.3）
- 不能"改代码"与"改约束"同一瞬间：旧代码可能仍写 `owner_id=NULL` → **两阶段发布/维护窗口**。
- 顺序：建 `user` → 插 migration user → 加 FK（仍 nullable）→ 回填 owner_id → **部署写 owner 的应用** → 验证无 NULL → 才加 NOT NULL/唯一约束。
- `CREATE UNIQUE INDEX CONCURRENTLY`（**不能在普通 alembic 事务里**，避免长锁）；FK 用 `NOT VALID` + 后续 `VALIDATE CONSTRAINT`。
- 换 `(owner_id,name)` 后**业务不可逆**（多用户同名后无法回退全局唯一）→ 回滚预案是**备份恢复**，不是假 downgrade。

### 8.4 认证工作量修正（补 §5）
- "~200 行"低估了：安全认证含哈希/登录/登出/当前用户/session 生命周期/禁用用户/限流/CSRF/错误语义/审计/测试。
- **自建适用**：邀请制、少量用户、邮箱密码——符合本项目轻依赖风格。
- **转折点**：一旦要开放注册/邮箱验证/忘记密码/OAuth/多设备 → 别继续自建，OAuth 单独引 `authlib`（不是 fastapi-users 全家桶）。

### 8.5 Basic Auth 边界（补 Phase A）
- agent/R-plumber/Postgres **绝不能被公网直连**：firewall + bind localhost/内网 **双保险**。
- Caddy Basic Auth 必须同时挡 `/api/*` 和 SPA 全路径。
- Basic Auth 阶段**不得承载真实多用户数据**（共享身份、无 ownership、无审计）→ 仅限演示/内测/评委。一旦让外部用户上传 PDF 或用平台 key，立即进 Phase B。

### 8.6 安全 checklist 补全（补 §4，公网前必过）
- [ ] **付费 API 滥用**：per-user 限流 + 并发限制 + 任务取消 + 每日额度 + 单任务 token/页数/文件数上限（即便 BYOK 也要防服务器被当代理打爆）
- [ ] **上传硬化**：文件名去信任化 + 随机路径 + 防路径穿越 + 解析超时 + 页数上限 + **zip/解压炸弹限制** + 后台任务资源上限
- [ ] **SSRF 覆盖重定向后最终地址**（不只初始 URL）+ 限协议/IP range + 防 DNS rebinding
- [ ] **日志脱敏**：Authorization / BYOK / OCR token / LLM base URL / `request_json` / `events_json` / error 全扫
- [ ] **FakeStreamClient**：公网无 key 时**不得静默产出假综述** → AI 功能显式 disabled/demo 标识
- [ ] **异步/下载/流接口鉴权**：stream / job polling / download / report 最易漏认证，全纳入权限矩阵
- [ ] **最小 admin 能力**：禁用用户 / 吊销 session / 查用量任务（无需完整后台，见 §9.3）
- [ ] CORS 上线移除 localhost/`*`（允许凭证时尤其危险）；备份含**恢复演练** + DB 与上传文件一致备份

### 8.7 BYOK 加固（补 §2.5）
Fernet 加密存 DB 只挡"DB 单独泄露"，不挡"应用服务器被打穿"。要求：
- master key **只来自环境/secret manager**，不进 DB/repo/镜像；设计 `key_version` 支持轮换。
- 明文只在调用瞬间存在，**永不写入** `request_json`/job/event/error；前端只显 masked fingerprint。
- 更强路径：**优先不存 BYOK**，让用户每次会话输入或浏览器本地存（体验差、风险低）。存 DB 是产品体验选择，非安全最优。

### 8.8 Phase B 验收标准（据二审升级）
从"有登录"改为：**所有资源入口都有作用域解析；所有异步/下载/流接口进权限矩阵；两个用户端到端互不可见。** 这比 session/JWT 选型重要得多。

### 9.2 积分计费系统（最小可用，不过度设计）
**分流**：请求带 `use_platform: bool`（或用户默认设置）。
- `use_platform=true` → 校验积分余额 → 用**平台 key** + 扣费；
- 否则 → 用**用户 BYOK key**，不扣费。

**数据表（新增）**：
```python
class RedeemCode(Base):          # 兑换码（充值积分）
    __tablename__ = "redeem_code"
    code: Mapped[str] = mapped_column(String(40), primary_key=True)
    credits: Mapped[int] = mapped_column(Integer)          # 面值
    used_by: Mapped[int | None] = mapped_column(ForeignKey("user.id"), nullable=True)
    used_at: Mapped[dt.datetime | None] = mapped_column(nullable=True)
    expires_at: Mapped[dt.datetime | None] = mapped_column(nullable=True)
    note: Mapped[str | None] = mapped_column(String(120), nullable=True)  # 批次/备注

class CreditLedger(Base):        # 积分流水（可审计，余额 = SUM(delta)）
    __tablename__ = "credit_ledger"
    id: Mapped[int] = _pk()
    user_id: Mapped[int] = mapped_column(ForeignKey("user.id", ondelete="CASCADE"), index=True)
    delta: Mapped[int] = mapped_column(Integer)            # +充值 / -消耗 / ±人工调账
    reason: Mapped[str] = mapped_column(String(24))        # redeem / consume / adjust / refund
    ref: Mapped[str | None] = mapped_column(String(64), nullable=True)  # code / ai_job.id
    balance_after: Mapped[int] = mapped_column(Integer)
    created_at: Mapped[dt.datetime] = mapped_column(server_default=func.now())
```
`User` 加 `credits: int default 0`（余额缓存，真值以 ledger 为准，二者一致性由事务保证）。

**原子扣费（防并发超扣，race condition）**：
```sql
UPDATE "user" SET credits = credits - :cost WHERE id = :uid AND credits >= :cost;
-- 影响行数 = 0 → 余额不足，拒绝任务
```
配合同事务写一条 `CreditLedger(delta=-cost, reason='consume', ref=job_id)`。

**异步任务计费**：ai_job 入队时**预扣**（按任务类型固定积分：如 parse=页数×K、review=N、chat=M，换算表进 config），失败/取消则写 `refund` 反向流水退回。事后可细化为按 LLM token 实际用量结算。

**兑换（用户端要做）**：Settings 页输入兑换码 → 事务内校验 `used_by IS NULL AND (expires_at IS NULL OR expires_at>now())` → 置 `used_by/used_at` + 写 `CreditLedger(delta=+credits, reason='redeem')` + 更新 `user.credits`。**并发同码只允许一次**（`UPDATE ... WHERE used_by IS NULL` 判定影响行数）。

## 11. Round 5 实施方案：全路由鉴权 + owner 隔离 + 不可逆迁移

### 影响评估
- **73 个业务路由**（main.py 68 + routes_research 5）几乎全是 `/projects/{pid}/...`，全部需鉴权 + owner 隔离。
- **27 个测试文件**调 `/projects` 端点，全部经统一的 `client` fixture（conftest:398）。
- `Project.name` 全局唯一（`uq_project_name`）→ 改 `(owner_id, name)`，**不可逆**。

### 策略（关键：测试适配只改 1 处）
1. **作用域依赖** `get_owned_project(pid, user) -> Project`：验证 project 存在且 `owner_id == user.id`，否则 404（不泄露存在性）。project-scoped 路由依赖它；子资源（corpus/paper/ai_job/agent_run/draft/artifact）经其所属 project 间接校验。
2. **认证注入**：project 路由挂 `get_owned_project`（内含 `get_current_user`）；无 project 的路由（/ai/ping、/sciverse/*、/library/stats）挂 `get_current_user`。**豁免**：/healthz、/auth/*、/ai/assets/*（静态）。
3. **测试适配（单点）**：`client` fixture 里 `app.dependency_overrides[get_current_user] = lambda: test_user` + 建一个测试用户 → 900 测试透明通过，无需逐个改。未认证测试单独写。
4. **owner 写入**：`create_project(owner_id=user.id)`；`list_projects(owner_id=user.id)` 只列自己的。
5. **不可逆迁移（单独一步，生产需用户执行）**：① 回填存量 project/paper 的 owner_id 到「迁移账户」② `Project.name` 唯一约束改 `(owner_id, name)` ③ `owner_id` NOT NULL。用 `CREATE UNIQUE INDEX CONCURRENTLY` + `NOT VALID`/`VALIDATE`；回滚靠备份。

### 分步（每步跑全量测试确认不破坏）
1. `get_owned_project` 依赖 + 仓储 owner 过滤（向后兼容：owner 可选）+ client fixture 注入认证。
2. 逐组路由挂依赖（corpus→analysis→ai→library→agent），每组后跑测试。
3. **权限矩阵测试**：user B 访问 user A 的 project/corpus/paper/ai_job/agent_run/下载/流 → 一律 404。
4. 不可逆迁移（测试库验证 apply；生产 apply 明确标注需用户备份后执行）。

### 风险
owner 过滤漏一处即跨租户泄露（§8.2 头号风险）→ 靠作用域依赖统一 + 权限矩阵全覆盖。迁移不可逆 → 生产前备份 + 测试库先验证。
