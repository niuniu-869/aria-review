# 文献综述生成系统 · 深度设计文档

> 目标:设计一个能产出**达到真实科研工作者标准**的文献综述的工具。
> 这里所说的"标准",不是"一篇读起来通顺的文章",而是一篇可以投稿到领域顶刊、
> 经得起审稿人质疑的**系统性文献综述(Systematic / Bibliometric / Narrative Review)**。
>
> 版本:v1.0 · 适用对象:BiblioCN(R-plumber + Python-FastAPI + TS-web 三服务)

---

## 0. 第一性原理:什么才算"真正的"文献综述

绝大多数 AI"综述工具"产出的东西,科研工作者一眼就能看穿是 slop。原因不在文笔,而在它们
**违反了文献综述的本质**。要做对,必须先把本质讲清楚。

### 0.1 文献综述不是"摘要的拼接",而是一个"论证"

一篇合格综述回答的是:**"关于 X 这个问题,人类目前知道什么、不知道什么、为什么。"**
它是一个**论证(argument)**,不是一份**清单(list)**。

- ❌ AI slop:"Smith(2019)研究了 A。Jones(2020)研究了 B。Lee(2021)研究了 C。"
  (author-centric 罗列,逐篇复述 = 摘要拼接)
- ✅ 真综述:"早期研究将 X 归因于 A 机制(Smith 2019; Jones 2020),
  但这一解释无法说明 Y 现象;近三年的工作转向 B 视角(Lee 2021; ...),
  二者的张力尚未解决——这正是本文识别的核心缺口。"
  (concept-centric 综合,有比较、有张力、有缺口)

这就是 Webster & Watson(2002, MIS Quarterly)提出的 **"concept-centric vs author-centric"** 黄金法则:
**好的综述围绕概念组织,而非围绕作者/论文组织。** 工具的全部设计都要服务于这一点。

### 0.2 三大支柱(恰好对应你强调的三件事)

| 你的要求 | 对应支柱 | 本质 | 失败时的样子 |
|---|---|---|---|
| **引用** | **可溯源性 Traceability** | 每一句论断都能回到具体原文 | AI 编造引用、张冠李戴、引用不支持论点 |
| **图表** | **结构化综合 Structural Synthesis** | 用文献计量把"场域全貌"可视化 | 配图是装饰、与正文无关、数据来路不明 |
| **脉络 + 分类总结** | **概念性综合 Conceptual Synthesis** | 梳理演化谱系 + 按主题归类提炼 | 按时间/作者流水账,无主题、无演化、无张力 |

一个工具要"像真人写的",这三根支柱必须**同时立住**,且**互相锚定**(正文↔图表↔原文三者可双向跳转)。

### 0.3 六条不可妥协的质量底线(贯穿全系统的硬约束)

1. **无证据不成句**:任何陈述性论断必须携带 ≥1 个可验证的源锚点(paper_id + 原文块),
   且系统在落稿前**逐句核验**该锚点的原文确实支持该论断(防 AI 幻觉)。
2. **综合而非复述**:章节生成强制走"主题—对比—张力—结论"结构,禁止"逐篇介绍"模板。
3. **可复现**:检索式、筛选标准、纳入/排除决策全程留痕,可一键生成 PRISMA 流程图与协议附录。
4. **图表即论证**:每张图由真实语料数据生成,带"图注 + 在正文中的解读句",而非贴图了事。
5. **人在环上(Human-in-the-loop)**:AI 做"脚手架与初稿",研究者在每个关键节点可审、可改、可否决。
6. **概念矩阵为骨**:在写作之前,先产出"概念×文献"编码矩阵(Concept Matrix),
   它是分类总结的结构化中间产物,也是综合写作的事实底座。

## 1. 真实科研工作者是怎么写综述的(参考流程,工具据此倒推)

设计工具前必须忠实还原真人流程。下面是管理学/会计/金融/医学等领域顶刊综述的**真实工作流**
(融合 PRISMA 2020、Webster & Watson 2002、Tranfield et al. 2003、Snyder 2019、
Paul & Criado 2020 的 SLR/SPAR-4-SLR 方法论):

```
阶段0 立项与协议(Protocol)
  └─ 提出研究问题 RQ → 定义纳入/排除标准 → 选库 → 设计检索式 → (医学领域注册 PROSPERO)

阶段1 检索与获取(Search & Acquisition)
  └─ 多库检索(WoS/Scopus/...) → 导出书目记录(BibTeX/RIS/WoS plaintext) → 获取全文 PDF

阶段2 筛选(Screening, PRISMA 四段漏斗)
  └─ 去重 → 题录(标题/摘要)筛选 → 全文资格审查 → 记录排除原因 → 产 PRISMA 流程图
     (顶刊常要求双人独立筛选 + Cohen's κ 一致性)

阶段3 数据抽取与编码(Extraction & Coding)
  └─ 逐篇抽取结构化字段(年份/期刊/理论/方法/数据/情境/构念/主要发现)
  └─ 建立编码方案 → 把每篇文献编码进"概念矩阵"(Concept Matrix / TCCM 框架)

阶段4 文献计量分析(Bibliometric Analysis,混合型综述必备)
  └─ 描述性:年度产出、高产期刊(Bradford)、高产作者/国家
  └─ 引用结构:高被引文献、历史引文网络 Historiograph(= 研究脉络/谱系)
  └─ 知识结构:共被引(知识基础/学派)、文献耦合(研究前沿)
  └─ 概念结构:关键词共现 → 主题地图(2×2:动力/基础/利基/新兴主题)、主题演化 Sankey
  └─ 社会结构:合著网络、三场图(国家—作者—关键词)

阶段5 综合与写作(Synthesis & Writing)
  └─ 主题综合:把文献聚成若干主题簇 → 每个主题内做"比较/对比/识别共识与张力"
  └─ 脉络叙事:这个领域如何演化?(奠基作 → 发展 → 当前前沿)
  └─ 识别缺口 → 提出未来研究议程(通常给一张 future agenda 表 / 概念框架图)

阶段6 引用与成稿(Citation & Manuscript)
  └─ 每个论断配引用 → 生成参考文献表 → 附"纳入研究汇总表"(Appendix)→ 格式化(APA/作者-年)
```

**关键观察(直接决定工具形态):**
- 真人写综述是一条**有评审检查点的流水线**,不是"一个按钮出全文"。工具必须把这条流水线显性化。
- **阶段3 的概念矩阵**是整篇综述的"事实数据库",分类总结和综合都从它派生——这是 AI 工具最常跳过、
  却最不该跳过的一步。
- **阶段4 的图**不是为了好看,每张都对应一种"结构"(知识结构/概念结构/社会结构/时间结构),
  是**用数据讲场域故事**。脉络梳理 = Historiograph(时间结构)+ 主题演化(概念结构随时间)。
- **阶段5 是综述的灵魂**,也是"像不像真人"的分水岭:综合(synthesis)≠ 总结(summary)。

## 2. 产品形态:一个"AI 增强的系统性综述工作台"

不是"输入主题 → 输出文章"的黑箱,而是一个**引导式、可溯源、人在环上的工作台(Workbench)**,
沿真实流程编排为若干阶段,每阶段有 AI 助手 + 人工检查点 + 结构化产物。

### 2.1 顶层信息架构

```
项目(Project)
 ├─ ① 协议  Protocol      研究问题、检索式、纳入排除标准
 ├─ ② 文献库 Library      书目记录 + 全文 + 解析后的结构化内容(可溯源单元)
 ├─ ③ 筛选  Screening     PRISMA 漏斗、纳入/排除决策与留痕
 ├─ ④ 编码  Coding        概念矩阵(主题 × 文献),分类的结构化底座
 ├─ ⑤ 分析  Analytics     文献计量图表(知识/概念/社会/时间结构)
 ├─ ⑥ 综述  Synthesis     大纲 → 逐章综合写作 → 逐句溯源核验
 └─ ⑦ 产出  Export        综述全文 + 图表 + 参考文献 + PRISMA + 汇总表 + 复现协议
```

### 2.2 核心数据契约:三层可溯源单元

整个系统的"可溯源"建立在把每篇文献拆成**三层粒度**之上(BiblioCN 的 MinerU + Track A 已具雏形):

```
Paper(文献)
  ├─ metadata     : 标题/作者/年份/期刊/DOI/被引/关键词/参考文献列表
  ├─ markdown     : MinerU 解析的全文(带版面)
  └─ blocks[]     : 结构化块(标题/正文段/表格/图/公式),每块带 {page, section, line_range}
        └─ 这是"原文锚点"的最小单位 —— 综述里每个引用都指向某个 block
```

> **设计要点**:综述正文中的引用不应只指向"Paper",而应尽量指向 **block**
> (某页某段某表),这样读者点击引用能直接跳到原文出处。这是与 AI slop 拉开差距的关键能力。

## 3. 功能模块详细设计(逐阶段)

### 模块① 协议与范围(Protocol)

**作用**:把"我想写个 X 综述"变成可执行、可复现的研究方案。这是综述科学性的起点。

**功能**:
- **研究问题向导**:引导用户写下 1–3 个 RQ(如"X 领域用了哪些方法?有哪些理论视角?未来方向?")。
  AI 可根据初始主题建议 RQ,但需用户确认。
- **检索式构建器**:可视化 Boolean 构造(AND/OR/NOT + 字段限定 + 时间窗 + 语言/文献类型),
  支持把同一概念的多种表述聚成一个"概念块"(如 `("text analysis" OR "textual analysis" OR NLP)`)。
- **纳入/排除标准(I/E criteria)**:结构化表单(年份范围、语言、文献类型、主题相关性、是否实证…),
  每条标准带唯一编号,后续筛选阶段排除文献时引用该编号 → 自动汇总到 PRISMA。
- **协议快照**:固化为一份可导出的 `protocol.md`,作为综述方法学章节与附录的原料。

**产物**:`Protocol{ research_questions[], search_strings[], inclusion_criteria[], exclusion_criteria[], databases[] }`

---

### 模块② 文献库与全文解析(Library & Parsing)

**作用**:把书目记录 + PDF 转成"可计算、可溯源"的结构化语料。

**功能**:
- **多源导入**:
  - 书目文件:WoS plaintext / Scopus CSV / RIS / BibTeX(供文献计量,含被引、参考文献列表)
  - 主题词检索:接 OpenAlex/Crossref 反查(BiblioCN 已有 `/corpus/from-topic`)
  - 参考文献反查:从一段参考文献文本抽取 → 反查元数据(已有 `/corpus/from-refs`)
  - PDF 上传:走全文解析(供编码与综合写作引用原文)
- **全文解析(关键)**:PDF → 结构化(MinerU)。产出 markdown + blocks,每块带 `{page, section, line_range, type}`。
  - ⚠️ **工程现实(本次压测实测)**:MinerU 云解析对学术 PDF(含公式/表格)**很慢**——
    实测 5 篇 >20 分钟。因此 **解析必须异步化**:导入即入库占位 → 后台任务队列解析 →
    前端轮询/推送进度,**严禁同步阻塞导入请求**(否则前端必超时)。详见 §8 工程改造项。
- **去重**:DOI 优先,辅以"标题归一化 + 作者 + 年份"模糊匹配,人工复核可疑对。
- **元数据回填**:解析出的标题/DOI 反查补全被引数、参考文献、关键词(文献计量需要)。

**产物**:`Paper{ id, metadata, markdown, blocks[], source_file, status }`

---

### 模块③ 筛选与 PRISMA(Screening)

**作用**:把"一大堆检索结果"收敛为"纳入研究集",并留下可复现的决策轨迹。

**功能**:
- **两阶段筛选**:① 题录(标题+摘要)粗筛 → ② 全文资格细筛。每篇打 `included / excluded / maybe`。
- **排除留痕**:排除时必须选一个"排除原因编号"(关联模块①的 E 标准)→ 自动累加到 PRISMA 计数。
- **AI 预筛(辅助,不替代)**:AI 依据 RQ 与 I/E 标准给出"建议纳入/排除 + 理由 + 置信度",
  研究者**复核确认**(可批量接受高置信、逐条审低置信)。AI 仅建议,决策权属人——这是方法学红线。
- **(可选)双人一致性**:支持两套独立筛选 + Cohen's κ 计算,满足顶刊"双盲筛选"要求。
- **PRISMA 流程图自动生成**:识别(各库命中数)→ 去重 → 题录筛 → 全文筛 → 纳入,
  每段数字由系统实时统计,一键出 PRISMA 2020 流程图(SVG/PNG)。

**产物**:`ScreeningDecision[]` + `PRISMA{ identified, duplicates, screened, excluded_by_reason{}, included }`

---

### 模块④ 编码与概念矩阵(Coding · 分类总结的骨架)

> 这是**最被 AI 工具忽略、却最决定"分门别类总结"质量**的模块。你强调的"分门别类对文献的总结",
> 其结构化底座就是这里的 **概念矩阵(Concept Matrix)**。

**作用**:在写作前,先把每篇纳入文献"读懂并归档"成结构化编码,形成"概念×文献"二维矩阵。
之后所有分类、综合、缺口识别都从这张矩阵派生 —— 它是综述的"事实数据库"。

**功能**:
- **编码方案(Codebook)**:用户定义维度,典型用 **TCCM 框架**(Theory 理论 / Context 情境 /
  Characteristics 构念变量 / Methodology 方法),也支持自定义维度(数据来源、样本、主要发现、局限)。
- **AI 辅助抽取**:对每篇文献全文,AI 按编码方案抽取各维度取值,**每个抽取值携带源 block 锚点**
  (这句结论来自第 5 页第 2 段)。研究者审核/修正,系统记录"人工已确认"。
- **主题归类(分类的核心)**:
  - 自上而下:用户预设主题类别,AI 把文献归类(可多标签)。
  - 自下而上:基于关键词共现/摘要聚类,AI 提议主题簇,用户命名与调整。
  - 二者可混合,最终形成"主题树"。
- **概念矩阵视图**:行=文献,列=主题/编码维度,单元格=该文献在该维度的编码值(带溯源)。
  支持按主题、年份、方法切片过滤 → 这就是"分门别类"的交互界面。

**产物**:`Codebook` + `CodedPaper[]` + `ConceptMatrix(themes × papers)` + `ThemeTree`

**为什么它让综述"像真人写的"**:有了概念矩阵,综合写作就能说出
"在'方法'维度上,主题 A 的 7 篇文献中有 5 篇用了 X 方法,而主题 B 普遍转向 Y"——
**这种横向比较只有结构化编码后才写得出来**,正是 AI slop 写不出的东西。

---

### 模块⑤ 文献计量分析(Analytics · 图表支柱)

**作用**:用定量方法刻画领域的四种结构,产出**作为论证的图表**。底层引擎:bibliometrix(R)。

**四类结构 → 对应图表(每张图都附"它在综述里回答什么问题"):**

| 结构 | 图表 | 回答的问题 | 在综述中的位置 |
|---|---|---|---|
| **描述性** | 年度产出趋势线 | 这个领域热度如何演变? | 引言/概览 |
| | 高产期刊(Bradford 定律) | 核心阵地在哪些期刊? | 概览 |
| | 国家/机构产出 | 谁在做这个研究? | 概览 |
| **时间结构** | **历史引文网络 Historiograph** | **领域的谱系:奠基作→关键节点→前沿** | **研究脉络章(核心)** |
| | 趋势主题(Trend Topics) | 哪些主题在兴起/衰退? | 脉络/未来方向 |
| **知识结构** | 共被引网络 | 有哪些"思想学派"/知识基础? | 理论脉络章 |
| | 文献耦合 | 当前研究前沿如何分群? | 主题章 |
| **概念结构** | **主题地图(2×2)** | **主题分四象限:动力/基础/利基/新兴** | **分类总结章(核心)** |
| | 主题演化 Sankey | 主题随时间如何分化/合并? | 研究脉络章 |
| | 关键词共现网络 | 概念之间如何关联? | 分类总结章 |
| **社会结构** | 合著网络 | 学术共同体协作格局? | 概览 |
| | 三场图(国家–作者–关键词) | 谁、在哪、研究什么的三元关系 | 概览 |

**功能要点**:
- 每张图由**当前纳入语料实时计算**(非静态贴图),参数可调(阈值/聚类数/时间切片),交互式(ECharts)。
- **图 ↔ 文双向锚定**:每张图配 `figure_id`,正文引用 `[图3]` 时建立链接;
  AI 生成"图注 + 解读句",研究者可改。点击图中节点(如 historiograph 的某文献)→ 跳到该 Paper。
- **脉络梳理的双图组合**:Historiograph(谁引谁,时间纵深)+ 主题演化 Sankey(主题怎么流变)
  = 完整的"研究脉络"可视化,直接喂给综合写作模块的"脉络叙事"章节。

**产物**:`Figure[]{ id, type, data, params, caption, interpretation, source_papers[] }`

### 模块⑥ 综述综合与写作(Synthesis · 灵魂)

**作用**:把协议、概念矩阵、文献计量结果,综合成一篇**有论证、有脉络、有分类、句句可溯源**的综述。
这是"像不像真人"的决胜模块。

**关键设计:不是"一键生成全文",而是"结构化大纲 → 逐章综合 → 逐句核验"三步。**

**6.1 大纲生成(Outline)**
- 基于主题树 + 文献计量结果,AI 提议综述骨架,研究者增删调序。典型结构(混合型综述):
  ```
  1. 引言(研究问题、为何重要、本综述贡献)
  2. 方法(检索式、PRISMA、纳入标准 —— 直接复用模块①③产物)
  3. 领域概览(文献计量描述性图表:产出/期刊/国家)
  4. 研究脉络(Historiograph + 主题演化:奠基→发展→前沿)   ← 脉络
  5. 主题综合(每个主题一节,概念矩阵驱动的分类总结)        ← 分类总结
     5.1 主题A:共识、分歧、张力、代表作
     5.2 主题B:...
  6. 研究缺口与未来议程(future agenda 表 + 概念框架图)
  7. 结论
  ```

**6.2 逐章综合写作(Synthesis, 非 Summary)**
- **每节注入"事实包"**:写主题 A 这一节时,系统从概念矩阵抽出该主题下所有文献的编码 +
  原文关键 block,作为该节的"证据上下文"喂给 LLM(而非把全文一股脑塞进去)。
- **强制综合结构的提示工程**:章节生成模板强制走
  **"主题立论 → 横向比较(谁同意谁/方法差异)→ 张力与分歧 → 小结与缺口"**,
  显式禁止"逐篇 A 做了…B 做了…"的罗列式输出。
- **引用即写即挂**:LLM 输出每个论断时,必须从"事实包"里挑出支撑它的 `paper_id/block_id` 作为锚点
  (结构化输出:`{sentence, claim_type, citations:[{paper_id, block_id, quote}]}`)。
- 研究者在所见即所得编辑器里改写、增删、调整;每句旁显示其引用锚点,可点开看原文。

**6.3 逐句溯源核验(Verification, 防幻觉的安全带)**
- 落稿前,系统对每个携带引用的句子跑**核验器**:把"句子 + 它声称的原文 block"交给一个独立 LLM 判定
  **"原文是否真的支持这句话?"**,输出 `supported / partial / unsupported / not_found`。
- `unsupported` / `not_found` 高亮告警,要求研究者修正或换引用 —— 把 AI 幻觉挡在成稿之前。
- (可由 §7 详述的可溯源系统具体实现。)

**产物**:`ReviewDoc{ outline, sections[]{ markdown, sentences[]{text, citations[]} }, provenance_map, verification_report }`

---

### 模块⑦ 产出与导出(Export)

**作用**:产出可直接用于投稿/汇报的完整产物集。

**功能**:
- **综述全文**:Markdown / DOCX / PDF,作者-年或编号引用格式(APA/GB-T-7714…),自动生成参考文献表。
- **图表附件**:所有文献计量图(高分辨率 PNG/SVG)+ PRISMA 流程图。
- **纳入研究汇总表(Appendix)**:由概念矩阵导出的"研究一览表"(作者/年/方法/主题/主要发现)。
- **复现协议**:检索式 + I/E 标准 + 筛选记录 + 软件版本 → 满足方法学可复现要求。
- **溯源报告**:每条引用 → 原文位置的映射表(可信度凭证)。

## 4. 可溯源引用系统(深挖:这是与 AI slop 的护城河)

引用是综述的命根子。AI 工具最致命的问题是**编造引用 / 引用不支持论点 / 张冠李戴**。
本系统用"四级溯源 + 落稿前核验"把这个问题从根上解决。

### 4.1 引用的四级粒度(越细越可信)

```
L1  指向 Paper           "(Smith 2019)"                  —— 最低要求,书目级
L2  指向 Section         "(Smith 2019, 方法部分)"         —— 章节级
L3  指向 Block           "(Smith 2019, 第5页表2)"         —— 段/表级,可一键跳原文
L4  指向 Quote span      高亮原文中支撑该论断的具体句子     —— 字句级,最强证据
```

> 系统默认追求 L3/L4。综述正文每个引用都能在阅读器里点开 → 右栏定位到 PDF/markdown 的具体块并高亮原句。
> BiblioCN 的 Track A(`build_block_line_ranges` + evidence resolver + `md_line` 锚点)已经做到 L3 雏形,
> 本设计把它扩展为 L4 并接入写作回路。

### 4.2 引用生成与核验的闭环

```
            ┌─────────────────────────────────────────────────────┐
写作时       │ LLM 输出一句论断,同时从"事实包"里选 block 作为锚点      │
(生成)       │   → {sentence, citations:[{paper_id, block_id, quote}]}│
            └───────────────────────┬─────────────────────────────┘
                                    ▼
            ┌─────────────────────────────────────────────────────┐
落稿前       │ 核验器(独立 LLM,只读原文 block,不看综述上下文)         │
(核验)       │   判定:原文是否支持该 sentence?                       │
            │   → supported / partial / unsupported / not_found     │
            └───────────────────────┬─────────────────────────────┘
                                    ▼
            ┌─────────────────────────────────────────────────────┐
告警         │ 非 supported → 高亮,要求人工修正/换引用/删句           │
            │ 生成"溯源可信度报告":全文 X 句,Y 句强支撑,Z 句存疑      │
            └─────────────────────────────────────────────────────┘
```

**要点**:核验器与写作器**职责分离、独立判断**(对抗式),避免"自己写自己批"的同源偏差。
这等价于给综述配了一个"内置审稿人"。

### 4.3 反"伪引用"的额外约束
- LLM 只能从**系统提供的事实包**里引用(白名单),不允许引用语料外的任何文献 → 杜绝凭空捏造 DOI。
- 参考文献表由"被正文实际引用且核验通过的 paper_id 集合"反向生成 → 不会出现"列了没引/引了没列"。

## 5. AI 与人的协作模型(各司其职 + 防幻觉)

**核心原则:AI 做"脚手架、抽取、初稿、核验",人做"判断、决策、定稿"。** 决策权永远在研究者。

| 阶段 | AI 的角色 | 人的角色(检查点) | 防幻觉机制 |
|---|---|---|---|
| 协议 | 建议 RQ / 检索式 | 确认、改写 | — |
| 筛选 | 预筛建议 + 理由 + 置信 | 复核、批准/否决 | 仅建议不决策 |
| 编码 | 抽取各维度 + 源锚点 | 审核、修正 | 每个抽取值带 block 溯源 |
| 分析 | 解读图、写图注 | 调参、采纳 | 图由真实数据算,非生成 |
| 写作 | 综合初稿 + 挂引用 | 改写、定稿 | 只能引白名单事实包 |
| 核验 | 逐句判原文是否支持 | 处理告警 | 独立对抗式核验器 |

**关键反模式(明令禁止):**
- ❌ "输入主题 → 一键出整篇" 的黑箱(无法溯源、必出幻觉、科研工作者不会信)
- ❌ AI 自动决定纳入/排除(方法学不可接受)
- ❌ 把整个语料全文塞进一个超长 prompt 让 LLM"自由发挥"(超窗、丢证据、罗列式输出)

## 6. 数据模型(核心实体)

```
Project { id, title, protocol_id, created_at }
Protocol { id, research_questions[], search_strings[], inclusion[], exclusion[], databases[] }

Paper {
  id, project_id, status(included|excluded|maybe|pending),
  metadata { title, authors[], year, journal, doi, cited_by, keywords[], references[] },
  markdown_text,
  blocks[] { block_id, type(title|text|table|figure|formula), page, section_title, line_range, text }
}

ScreeningDecision { paper_id, stage(title_abstract|full_text), decision, exclusion_reason_id, reviewer, note }
PRISMA { identified_by_db{}, duplicates_removed, screened, excluded_by_reason{}, eligible, included }

Codebook { dimensions[] { key, label, type } }           // 如 TCCM
CodedPaper { paper_id, codes{ dimension_key: { value, source_block_id } }, themes[] }
ConceptMatrix { themes[], papers[], cells{ (theme,paper): coded_value } }

Figure { id, type, params, data, caption, interpretation, source_paper_ids[] }

ReviewDoc {
  id, outline[],
  sections[] {
    title, markdown,
    sentences[] { text, claim_type, citations[] { paper_id, block_id, quote, verify_status } }
  },
  references[],            // 由实际引用反向生成
  verification_report { total, supported, partial, unsupported }
}
```

## 7. 与现有 BiblioCN 的映射(已有什么 / 要补什么)

本设计**不是推倒重来**,而是把 BiblioCN 已有能力串成上述流水线。盘点:

| 模块 | 现状(已具备) | 需补强 |
|---|---|---|
| ① 协议 | 部分(主题词/参考文献入口) | 检索式构建器、I/E 标准结构化、协议快照 |
| ② 文献库/解析 | ✅ MinerU 全文解析、blocks、import | **解析异步化**(见下)、L4 quote 锚点、去重 |
| ③ 筛选/PRISMA | ✅ 已有 `/prisma`、纳入排除状态 | 排除原因编号化、AI 预筛、双人 κ |
| ④ 编码/概念矩阵 | ❌ **基本缺失(最大缺口)** | Codebook、AI 抽取、概念矩阵视图、主题树 |
| ⑤ 文献计量 | ✅ bibliometrix 全套图(overview/historiograph/thematic/coupling/three-field…) | 图↔文锚定、图注 AI 解读、节点点击跳转 |
| ⑥ 综述写作 | ✅ ai/jobs `run_review` + 模板 + provenance_map(Track A) | 概念矩阵驱动的"事实包"、强制综合结构、逐句核验 |
| ⑦ 导出 | 部分(综述/PRISMA/图) | 概念矩阵→汇总表、复现协议、溯源报告 |

### 7.1 必做的工程改造:导入/解析异步化(P0,压测实证)

> 本次压测实测:`papers/import` **同步调用 MinerU 云解析**,5 篇 PDF >20 分钟未返回,
> 全量 66 篇需数小时,前端必然超时。这是当前架构的**头号瓶颈**,也是上线大语料前的硬阻塞。

**改造方案**:
```
现状(同步,阻塞):
  POST /papers/import ──同步──> 上传OSS ──同步等待──> MinerU解析(分钟级) ──> 入库 ──> 返回
  └─ 5篇>20分钟,前端超时

目标(异步,任务化):
  POST /papers/import ──> 立即入库占位(status=parsing) ──> 返回 job_id(秒级)
                          └─> 后台任务队列:逐篇/批量送 MinerU ──> 解析完回填 ──> status=parsed
  GET  /papers/import/jobs/{id} ──> 进度(已解析 N/总 M, 失败列表)
  前端:进度条 + SSE/轮询,解析完的文献先可用,失败可重试
```
- 加并发上限与重试退避,记录每篇耗时/失败原因 → 可观测。
- 大批量(如 66 篇)走"提交即返回 + 后台异步解析 + 完成通知"。

### 7.2 第二大缺口:概念矩阵(模块④)

这是把"图表 + 文章"升级为"真正的分类综述"的关键中间层,目前 BiblioCN 完全没有。
它让综述具备"横向比较"能力(分门别类总结的前提),建议作为 P0 与异步化并列。

## 8. 质量验收标准(怎么判断产出"够格")

一篇产出能不能叫"达到科研工作者标准",用下面这张 checklist 自检(也可做成系统内置评分卡):

**可溯源性**
- [ ] 全文每个陈述性论断都有 ≥1 引用锚点
- [ ] 引用核验报告中 `unsupported/not_found` = 0(或已人工处理)
- [ ] 参考文献表与正文引用完全一致(无引而未列/列而未引)

**概念性综合(非复述)**
- [ ] 章节按**主题**组织,而非按作者/时间流水
- [ ] 每个主题节包含:共识 + 分歧/张力 + 缺口(不是逐篇介绍)
- [ ] 有明确的研究脉络叙事(奠基→发展→前沿)

**结构化与分类**
- [ ] 存在概念矩阵,且每篇纳入文献都已编码
- [ ] 有"纳入研究汇总表"附录

**图表即论证**
- [ ] 每张图来自真实语料数据,有图注 + 正文解读句 + 在正文被引用
- [ ] 至少覆盖:PRISMA、年度产出、Historiograph(脉络)、主题地图(分类)

**可复现**
- [ ] 检索式、I/E 标准、PRISMA 计数、筛选记录可导出

## 9. 实施路线图(分期,先立支柱再加润色)

**P0 — 打通"可溯源 + 异步"地基(上线前必做)**
- 导入/解析异步化(§7.1);Track A 引用溯源接入写作回路;逐句核验器(§4.2)。

**P1 — 立起"分类"支柱**
- 概念矩阵模块④:Codebook + AI 抽取(带溯源)+ 概念矩阵视图 + 主题树。
- 综合写作改造:概念矩阵驱动的"事实包" + 强制综合结构提示。

**P2 — 立起"脉络 + 图表"支柱**
- 文献计量图↔文锚定、图注 AI 解读、节点点击跳原文。
- Historiograph + 主题演化双图驱动"研究脉络"章自动初稿。

**P3 — 成稿与方法学完备**
- PRISMA 排除原因编号化、(可选)双人 κ;复现协议导出;汇总表/溯源报告;引用格式化(APA/GB-T-7714)。

**P4 — 打磨**
- 多型综述模板(叙事/系统/文献计量/范围综述)、协作多人评审、版本对比。

## 10. 风险与开放问题

- **MinerU 速度/成本**:云解析慢且按量计费。需评估本地化部署或缓存层;异步化只是缓解体验,不降总耗时。
- **AI 综合质量天花板**:即使有事实包,LLM 仍可能产"正确但平庸"的综合。需靠强模板 + 人工定稿兜底,
  并对"张力/缺口识别"这类高阶判断保持人主导。
- **概念矩阵的编码一致性**:AI 抽取需要人工校准,大语料下的审核成本不可忽视(可做主动学习/抽样复核)。
- **引用核验的误判**:核验器自身也可能错判 partial/unsupported;需给人工"覆盖核验结论"的出口。
- **不同学科的差异**:医学重 PRISMA/meta,管理学重 TCCM/主题综合,人文偏叙事。模板需可配置,勿一刀切。

## 附:一句话总结设计立场

> **真正的文献综述工具,不是"把文献喂给 AI 让它写一篇",
> 而是"用 AI 把研究者从机械劳动里解放出来,同时强制保证每一句话都可追溯、每一张图都讲道理、
> 每一类文献都被结构化地比较过"。** 工具的价值在于守住科研的严谨底线,而不是替研究者思考。
